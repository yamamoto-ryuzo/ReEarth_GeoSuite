# レイヤ階層化・排他グループ機能 仕様書

## 1. 概要
Re:Earth GeoSuite プラグインにおいて、レイヤリストを階層的（ツリー状）に表示し、グループごとの一括制御や排他制御（ラジオボタン動作）を提供する機能の仕様について記述する。

## 2. グループの定義方法

レイヤのグループ化は、レイヤが持つメタデータまたはタイトル文字列を解析して行われる。

### 2.1. データ優先 (`layer.data.group`)
Re:Earthのレイヤデータ (`layer.data.group`) にパス文字列が設定されている場合、これを最優先で使用して階層構造を構築する。
ウィジェットやデータセット側で明示的にグループを指定する場合に推奨される。

### 2.2. タイトル解析 (Fallback)
`layer.data.group` が未設定の場合、レイヤタイトル (`layer.title`) に含まれるスラッシュ (`/`) を区切り文字として解析し、階層構造を構築する。
*   例: タイトルが `東京都/千代田区/建物` の場合
    *   第1階層: `東京都`
    *   第2階層: `千代田区`
    *   レイヤ名: `建物`

## 3. グループの種類と記法

パスの区切り文字によって、グループの挙動（通常または排他）が決定される。
同名のグループであっても、区切り文字の種類が異なれば、システム内部では「別のグループ」として扱われ、ツリー上でも分離して表示される。

| 種類 | 記法 | 例 | 概念 |
| :--- | :--- | :--- | :--- |
| **通常グループ** | `/` (シングルスラッシュ) | `親/子` | フォルダ。複数選択可能。 |
| **排他グループ** | `//` (ダブルスラッシュ) | `親//子` | ラジオボタン。グループ内で**1つのみ**選択可能。 |

## 4. 詳細動作仕様

### 4.1. 通常グループ (`/`)
フォルダのような振る舞いをする標準的なグループ。

*   **親（グループ）チェックボックス操作**:
    *   **ONにする**: グループに含まれる**すべての**子レイヤ・孫レイヤに対して `show` (表示) メッセージを送信する。
    *   **OFFにする**: グループに含まれる**すべての**子レイヤ・孫レイヤに対して `hide` (非表示) メッセージを送信する。
*   **子レイヤ操作**:
    *   個別にON/OFFの切り替えが可能。
    *   ある子レイヤを操作しても、兄弟レイヤ（同じグループ内の他のレイヤ）の表示状態には影響しない。
    *   **親へのUI連動**: 親グループがOFF（未チェック）の状態で、ある子レイヤをONにした場合、親グループのチェックボックスを**UI上のみ**ON（チェック状態）に変更する。
        *   *注: この際、親グループとしての `show` メッセージは送信されない（他の子はOFFのまま維持される）。*

### 4.2. 排他グループ (`//`)
ラジオボタンのような振る舞いをする、選択式のグループ。

*   **親（グループ）チェックボックス操作**:
    *   **ONにする**: グループ内の**一番上（リスト先頭）の子レイヤのみ**を表示 (`show`) し、それ以外の子レイヤをすべて非表示 (`hide`) にする。
    *   **OFFにする**: グループ内のすべての子レイヤを非表示 (`hide`) にする。
*   **子レイヤ操作**:
    *   **ONにする**: 選択したレイヤを表示 (`show`) し、**同じグループ内の他のすべての兄弟レイヤを自動的に非表示 (`hide`)** にする。
*   **初期化時の強制挙動**:
    *   プラグイン読み込み時（初期化時）に、排他グループ内で「表示 (Visible)」になっているレイヤを確認する。
    *   **誰もONでない場合**: 強制的にリストの一番上の子レイヤをONにする。
    *   **複数がONの場合**: リストの一番上のONレイヤを残し、他を強制的にOFFにする。
    *   これにより、排他グループは（親がOFFでない限り）「常にどれか1つだけが選ばれている」状態を維持する。

### 4.3. レイヤのリフレッシュ処理と待機要件

レイヤの表示/非表示（ON/OFF）操作は非同期なリフレッシュ処理を伴うため、直後に続けて別の切替や再読み込みを行うと競合や表示の不整合が発生する可能性がある。実装では以下のいずれかの方法で待機を確保することを推奨する。

- レイヤのロード完了・アンロード完了を示すイベント（例: `loaded` / `unloaded`）やコールバックを待ってから次の操作を許可する。
- 明示的なイベントが無い場合は、操作間に短い遅延（例: 300〜500ms）を挟むか、操作をデバウンスして連続処理を抑制する。
- 複数レイヤに対する一括操作では、個別リクエストを直列化して順に実行し、各リクエスト完了を待ってから次へ進める。

UI上は操作中にスピナーや無効化（グレーアウト）等のフィードバックを表示し、ユーザーに処理待機が必要であることを明確に示すこと。

この待機仕様は、特に外部データソース（タイルサーバや3D Tiles等）を使うレイヤの追加・削除・再読み込み時に重要である。

## 5. UI/UX仕様

ユーザーがグループの種類を直感的に判別できるよう、視覚的な差異を設ける。

### 5.1. 通常グループ
*   **表示**: 標準的なツリー表示。
*   **チェックボックス**: 標準的な四角いチェックボックス (☐)。

### 5.2. 排他グループ
*   **グループ名表示**: グループ名の後ろに `⊚` (二重丸) アイコンを表示し、特別なグループであることを示す。
*   **子レイヤの表示**:
    *   子レイヤのチェックボックスを CSS で**円形（ラジオボタン風）**にスタイリングする。
    *   これにより、「この中から1つしか選べない」というアフォーダンスを提供する。
